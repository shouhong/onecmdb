The Template Hierarchy

The tree view in the leftmost pane displays all Templates in the current OneCMDB model. Note that this includes user defined templates as well as model specific templates and system (internal) templates. As you can see, all models start with a Root Template From here and down all templates are structured in an object-oriented (class-oriented) fashion. This means that the Root Template contains a few, generic attributes. Templates on the level below the root inherit all attributes from it, and so in. A template (child) always inherits from the template above it (parent). For each level down the templates become more and more specialized and rich. For example, new attributes can be added, new references created or attribute values can be filled in.

The number within parenthesis behind each template name in the tree view states how many instances that exists on this level. I.e. based on this level´s template, or any other template (child) further down in the tree. If you click on a template a table is displayed showing general info, attributes, references and instances based on this template. If you click on the tiny down-arrow besides the template name you get a submenu with choices for creating a new child template, a new instance based on this template or to delete the current template.

The template names may not be that easy to understand. Here is a brief introduction.

  Name Short explanation  

 

Account

Here you find OneCMDB login accounts

 

 

CI

Here you find all templates and instances that are CI:s

 

 

Job

Here you find system related information that relate to jobs in OneCMDB

 

 

OneCMDB Config

Here you find system related information that relate to the OneCMDB configuration

 

 

Policy

Here you find system related information that relate to policies in OneCMDB

 

 

Reference

Here you find all templates and instances that are References (for instance between CI:s)

 

 

Service Management

Here you find things specific for a few of the user applications

 

 

Transform

Here is system information used when for instance transforming import files

 

Hence, the CI section is a good place to start. Avoid the system related sections unless you are really sure what you are doing.
